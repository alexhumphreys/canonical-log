# canonical-log

Stripe-style canonical log lines for JVM services. Pulling in `canonical-log-spring-boot-starter` (or a per-library module) auto-attaches HTTP, DB, and other observability fields to a single log line emitted at the end of each unit of work.

The POC works end-to-end on a Spring Boot sample app. This document captures the architecture, the decisions behind it, and the work queue.

## Architecture

```
                    ┌─────────────────────────────────────┐
                    │  WorkUnit (HTTP request, Kafka      │
                    │  consume, scheduled job, ...)       │
                    └──────────────┬──────────────────────┘
                                   │ start
                                   ▼
                    ┌─────────────────────────────────────┐
                    │  CanonicalLogContext                │
                    │  (ConcurrentHashMap accumulator)    │
                    └──────────────┬──────────────────────┘
                                   │ bound to thread/coroutine via
                                   │ ThreadContextElement bridge
                                   ▼
        ┌──────────────┬───────────┴──────────┬──────────────┐
        ▼              ▼                      ▼              ▼
  ┌──────────┐  ┌──────────┐          ┌──────────────┐  ┌─────────┐
  │ HTTP     │  │ JDBC     │   ...    │ Handler code │  │ Future  │
  │ adapter  │  │ adapter  │          │ (markFailed, │  │ contrib │
  │          │  │          │          │  custom flds)│  │ -utors  │
  └────┬─────┘  └────┬─────┘          └──────┬───────┘  └────┬────┘
       │             │                       │               │
       └─────────────┴───────────────────────┴───────────────┘
                                   │ contribute fields
                                   ▼
                    ┌─────────────────────────────────────┐
                    │  emit on WorkUnit completion        │
                    │  → flat JSON, snake_case, one line  │
                    │  → stdout / Loki / etc.             │
                    └─────────────────────────────────────┘
```

The accumulator lives for the lifetime of one work unit. Contributors (libraries that hook into native extension points) and the handler both write into it. Exactly one canonical log line is emitted at the end. That's the whole product.

## Module layout

- `canonical-log-core` — accumulator, `WorkUnit`, `WorkUnitAdapter`, `Outcome`, `CanonicalLogContext`, the `ThreadContextElement` bridge, the `CanonicalLogMdc` MDC mirror (see the gotchas entry), `CanonicalFields` (the published `const` names for every field the library writes — see the field-constants gotcha), the `openCanonicalWorkUnit`/`CanonicalWorkUnitScope` blocking open/close primitive (delicate; see the open/close-primitive gotcha), and the two framework-free sink seams every integration needs — `CanonicalLineWriter` (the named injectable sink; `EmitFn` stays the raw lambda primitive the lifecycle takes, and entry points bridge them with `scope.emit(writer::write)`) and `CanonicalLogSampler` (the post-enrich emit predicate) — plus `MdcCanonicalLineWriter`, a default-capable sink that flattens the snapshot into slf4j MDC for one event (stringifying every value) for JSON layouts that render MDC but not logstash `StructuredArguments` (Dropwizard's `EventJsonLayout`); prefer the logstash writer when the encoder is present because it preserves field *types*, and never wire both (todo 020). Also `JsonCanonicalLineWriter` + the public `canonicalLineJson(fields)` serializer (todo 028) — a dependency-free typed sink whose emitted **message is a hand-rolled flat JSON object** (keys sorted, own RFC-8259 escaping, the accumulator's type rules kept intact: `Long`/`Boolean` bare, `NaN`/`Infinity` as strings, enums as `name`, everything else `toString()`-as-string, a throwing `toString()` caught per-entry as `"<serialization_failed: FQCN>"`, never throws). The human `message` is folded *inside* the object under a `message` key (check-before-default: not clobbered if the handler set one), and an `emitLine` lambda decouples transport (default `logger.info(String)` — the single-arg overload that never does `{}` substitution, safe with braces). **Sink chooser: logstash encoder configured → `LogstashCanonicalLineWriter`; MDC-flattening layout → `MdcCanonicalLineWriter`; dedicated raw appender / anything else → `JsonCanonicalLineWriter`.** Framework-agnostic (still `slf4j-api` + coroutines only; the round-trip property test's Jackson oracle is `testImplementation` only). The seam types were moved here from the umbrella starter in todo 020; deprecated typealiases remain in the `...canonicallog.spring` package for one release.
- `canonical-log-okhttp` — OkHttp `Interceptor` that contributes `http_client_request_count`, `http_client_request_duration_ms_total`, plus the `Request.Builder.withCanonicalContext()` tag helper that extends contribution to the async `enqueue()` path (see the OkHttp `enqueue()` gotcha). Framework-agnostic.
- `canonical-log-jdbc` — `datasource-proxy` `QueryExecutionListener` that contributes `db_query_count`, `db_execution_count`, `db_execution_duration_ms_total`, `db_slow_execution_count`, `db_execution_error_count`. Framework-agnostic.
- `canonical-log-logstash` — the default sink, `LogstashCanonicalLineWriter`, split out of the umbrella starter in todo 020 so non-Spring integrations (servlet, Dropwizard) and the scheduling starter can share one writer without depending on the servlet umbrella. Logs the snapshot as logstash-logback-encoder `StructuredArguments` on the `"canonical"` logger, preserving field types; the encoder is an `implementation` dep (never in public signatures). `api(project(":canonical-log-core"))`. Framework-agnostic — adopters whose layout doesn't use the encoder simply don't depend on it (they use core's `MdcCanonicalLineWriter` or their own).
- `canonical-log-servlet` — framework-neutral servlet HTTP integration (todo 021), extracted from the umbrella starter so any `jakarta.servlet` stack (Jetty/Dropwizard, plain Tomcat, Jersey-on-servlet) gets the hard-won request lifecycle without re-deriving it. Holds: `HttpExchange` + `HttpWorkUnitAdapter` (the status/error/499-cancel heuristics; route lookup is now a constructor param `routeResolver`, defaulting to reading the `HttpWorkUnitAdapter.ROUTE_ATTRIBUTE` request attribute — any glue that publishes the matched route there gets `http_route` for free); `runCanonicalHttpRequest(...)`, the `@DelicateCanonicalLogApi` entry-point helper that owns the whole open → chain → sync/async emit split → unbind sequence plus the emit-exactly-once `CanonicalLogAsyncEmitListener` and `AsyncTimeoutCancellationException` (both internal) — **this is the one copy of the lifecycle both filters call**; `CanonicalLogServletFilter`, a plain `HttpFilter` caller (writer has no default — deployment-specific — plus a once-per-request attribute guard and an `includeRequest` predicate); and `PathExclusions.matcher(...)`, a two-rule (exact / trailing-star prefix) dependency-free path excluder. `api(project(":canonical-log-core"))`, `compileOnly` servlet-api **5.0.0** (the lowest `jakarta.*` servlet API, so it loads on EE9 *and* EE10+ containers; the pin is the audit that no 6.x-only API creeps in). Zero Spring dependencies. The moved `HttpExchange`/`HttpWorkUnitAdapter` leave `@Deprecated` typealiases in `...canonicallog.spring` for one release (they appear in adopter `WorkUnitAdapter<HttpExchange>` beans).
- `canonical-log-dropwizard` — Dropwizard 4 bundle (todo 022) — the last-mile glue that turns a Dropwizard service (Jersey-on-Jetty) into a canonical-log adopter with one `bootstrap.addBundle(CanonicalLogBundle())`. `CanonicalLogBundle` (a `ConfiguredBundle<Configuration>`, constructor-configured — adapter/writer/sampler/excludePaths defaults, no builder) registers `CanonicalLogServletFilter` on the **application** servlet environment (`/*`, `REQUEST` dispatch only — the filter's once-per-request guard covers re-dispatches) and `JerseyRouteCaptureFilter` on the Jersey environment; nothing on the **admin** environment, so admin-port healthchecks/metrics never open a work unit (app-port health endpoints that shouldn't emit need `excludePaths`). `JerseyRouteCaptureFilter` is a post-matching (deliberately **not** `@PreMatching`) `ContainerRequestFilter` that reads `ExtendedUriInfo.getMatchedTemplates()` (most-specific-first; the full route is the **reversed** join with `//`→`/` normalization — pinned empirically with a sub-resource locator in `CanonicalLogBundleTest`) and publishes it to `HttpWorkUnitAdapter.ROUTE_ATTRIBUTE`, where the servlet adapter's default `routeResolver` picks it up as `http_route`; its whole body is try/catch-WARN (route capture is telemetry, never 500s a request; a 404 before matching simply omits `http_route`, matching Spring). Default writer is core's **`MdcCanonicalLineWriter`** (not logstash): Dropwizard's stock `EventJsonLayout` renders MDC but not logstash `StructuredArguments`, so canonical fields ride MDC and appear as top-level JSON keys under `layout: { type: json, flattenMdc: true }` (the documented trade-off: MDC stringifies values — counts/durations arrive as strings; adopters wanting typed fields pass `LogstashCanonicalLineWriter` from `canonical-log-logstash` and route the `"canonical"` logger to a logstash-encoder appender). `work_unit_id` MDC correlation on handler logs works with no extra wiring (the servlet filter installs it); the process-wide `CanonicalLogMdc.enabled` switch is the opt-out (no bundle flag). `api(project(":canonical-log-servlet"))` (brings core transitively), `compileOnly(libs.dropwizard.core)` (4.0.x, jakarta namespace — the adopter's Dropwizard version wins at runtime; Dropwizard 2/3 `javax.*` unsupported; Jersey types like `ExtendedUriInfo` come in transitively via dropwizard-jersey, no explicit jersey-server dep). No Spring types anywhere. Error-semantics finding: with Dropwizard's default exception mappers on, Jersey maps an uncaught resource exception to 500 *before* the filter unwinds, so the line gets `error=true`/`error_reason=server_error` (completed-5xx heuristic) but **no** `error_class`; disabling default mappers (as the e2e test does, to exercise the path) lets the throw reach the filter as `Outcome.Threw`, adding `error_class` (the container's wrapper, `jakarta.servlet.ServletException`). Tested with `DropwizardTestSupport` (in-process Jetty, no Docker) driven from a kotest `DescribeSpec` — `support.before()`/`after()` in `beforeSpec`/`afterSpec`, appender attached to the `"canonical"` logger *after* boot (the logback-init gotcha).
- `canonical-log-tracing-otel` — trace correlation (todo 027): `OtelSeedingAdapter<T>`, a `WorkUnitAdapter` decorator (by-delegation, package `io.github.alexhumphreys.canonicallog.otel`) that reads `Span.current().spanContext` in `seed` (after `delegate.seed`) and writes `trace_id`/`span_id` **only when `spanContext.isValid`** — never OTel's all-zeroes sentinel. `seed`, not `enrich`, is the whole point: the current span is ambient opening-thread state, gone by `enrich` time on an async listener / `onStop` thread (the `OtelSeedingAdapterTest` async pin — a suspend unit that `withContext(Dispatchers.IO)`s with the span ended right after open — fails if the logic moves to `enrich`). No span creation, no baggage, no context manipulation (baggage-to-fields is operator cardinality policy, an adopter-side `seed`). `api(project(":canonical-log-core"))`, `compileOnly(libs.opentelemetry.api)` (only `Span.current()`/`SpanContext`, stable since OTel 1.0, so the adopter's OTel version wins at runtime; tests use `opentelemetry-sdk-testing`'s `SdkTracerProvider`). This is the shipped shape of the old "Micrometer tracing contributor" v0.2 item — Micrometer users get it via their OTel bridge. **Core stays OTel-free** (the anti-goal): the zero-dependency companion is core's `MdcSeedingAdapter<T>` (a `WorkUnitAdapter` decorator taking a `Map<String,String>` of MDC key → canonical field, copied in `seed`; `put` drops nulls so absent keys cost nothing) for agent/framework stacks that already mirror ids into MDC (`trace_id`, `dd.trace_id`, `traceId`). Both are the shipped composition idiom from `WorkUnitAdapter.seed`'s KDoc (todo 023); `TRACE_ID`/`SPAN_ID` are `CanonicalFields` constants but written by these seeding modules, not core.
- `canonical-log-okhttp-spring-boot-starter` — auto-config providing an `OkHttpClientBuilderCustomizer` bean that adopters apply to their own `OkHttpClient.Builder` constructions. **Unlike the HTTP filter and JDBC starters, this one requires adopter participation** because `OkHttpClient` is configured at builder time rather than at bean construction. Opt-out: `canonical-log.okhttp.enabled=false`.
- `canonical-log-jdbc-spring-boot-starter` — auto-config wiring the JDBC contributor (uses `@JvmStatic` companion-object `@Bean` for the BeanPostProcessor per Spring's recommendation).
- `canonical-log-spring-boot-starter` — umbrella starter that pulls in the others plus the HTTP request adapter (servlet filter that creates the work unit, captures `http_request_method` / `http_route` / `http_response_status_code` / `http_request_duration_ms`, emits the line). Since todo 021 the request lifecycle itself lives in `canonical-log-servlet` (pulled in via `api`): the Spring `CanonicalLogFilter`'s whole `doFilterInternal` body is one call to `runCanonicalHttpRequest`, and it stays `OncePerRequestFilter` only for the Spring-specific bits — its registration semantics, the `AntPathMatcher` `exclude-paths` (back-compat), and its default adapter's route resolution via `springRouteResolver` (reads Spring MVC's `BEST_MATCHING_PATTERN_ATTRIBUTE` first, then the servlet-generic `ROUTE_ATTRIBUTE`). **The auto-config builds the default adapter as `HttpWorkUnitAdapter(springRouteResolver)` — a bare `HttpWorkUnitAdapter()` under Spring would silently lose `http_route` (it'd only read `ROUTE_ATTRIBUTE`).** The filter's collaborators are injectable seams: a user `WorkUnitAdapter<HttpExchange>` bean replaces the default `HttpWorkUnitAdapter` (compose with it, don't subclass — delegate describe/enrich, then put extra fields), and a user `CanonicalLineWriter` bean replaces the default logstash sink (`LogstashCanonicalLineWriter`, now from `canonical-log-logstash`, pulled in via `api`). The **adapter and sampler** resolve via `ObjectProvider` — deliberately not `@ConditionalOnMissingBean` on those default beans, because CoMB matches raw types and an adapter bean for a different work-unit type would wrongly suppress the HTTP default. The **writer** is the exception (todo 020): `CanonicalLineWriter` has no type parameter, so the default is a real `@Bean` guarded by `@ConditionalOnMissingBean(CanonicalLineWriter)` and injected straight into the filter. That default bean is *shared with the scheduling starter* (see its entry), so one user writer bean overrides the sink for HTTP **and** scheduled-job lines. Two knobs control which requests produce a line: `canonical-log.http.exclude-paths` (ant-style patterns matched against `request.requestURI` — consistent with `url_path`; matching requests skip the work unit entirely via `shouldNotFilter`, so probes like `/actuator/**` cost nothing) and a user `CanonicalLogSampler` bean (also resolved via `ObjectProvider`, default emit-all; consulted *after* `adapter.enrich` so the predicate sees status/duration/error and can implement "always keep errors, sample healthy 200s at 1%"). No rate-based built-in sampler yet — the hook is the feature. Enrich/sampler/writer failures are swallowed and WARN-logged rather than failing the request (see the "telemetry failures never replace the block's result" gotcha); a throwing sampler fails open, so the line is still written.
- `canonical-log-scheduling-spring-boot-starter` — instruments `@Scheduled` methods transparently (no wrapping in the job body) by hooking Spring's **own scheduled-task observation** rather than AOP: it registers a `CanonicalScheduledTaskObservationHandler` (an `ObservationHandler<ScheduledTaskObservationContext>`) on an `ObservationRegistry` and wires that registry onto the `ScheduledTaskRegistrar` via a `SchedulingConfigurer`. Spring runs each scheduled task as `observation.observe(runnable)`, whose `onScopeOpened → [method] → onScopeClosed → onStop` callbacks map onto the core `openCanonicalWorkUnit`/`CanonicalWorkUnitScope` open/close primitive (bind at scope-open so contributors on the scheduler thread land, unbind at scope-close, enrich+emit at stop from `context.error`). Default `job_name` = `<SimpleClassName>.<method>`, `work_unit_kind=scheduled_job`, via `ScheduledJobWorkUnitAdapter` (override with a `WorkUnitAdapter<ScheduledTaskObservationContext>` bean). No separate framework-agnostic contributor module — scheduling instrumentation is inherently Spring (the observation context + handler are Spring/Micrometer types), so it lives entirely in the starter, like the HTTP filter does. Opt out with `canonical-log.scheduling.enabled=false`. Requires `@EnableScheduling` as usual; needs no actuator (it provisions a private `ObservationRegistry` when the app has none). The line is written through a `CanonicalLineWriter` bean, not a hardcoded logstash emit (todo 020): the starter registers its own default `LogstashCanonicalLineWriter` guarded by `@ConditionalOnMissingBean(CanonicalLineWriter)` (dep: `canonical-log-logstash`) so it works standalone, and declares `@AutoConfiguration(afterName = ["...spring.CanonicalLogAutoConfiguration"])` (string class name, so no compile dependency on the umbrella starter) so that when both starters are present the umbrella's default writer registers first and this one backs off — exactly one default regardless of which starters are on the classpath. A user `CanonicalLineWriter` bean therefore overrides the sink for scheduled-job lines too (pinned by `CanonicalSchedulingWriterOverrideTest`).
- `canonical-log-kafka` — Kafka integration (todo 026), framework-agnostic, `compileOnly(libs.kafka.clients)` (only `ConsumerRecord`/`Producer`/`Callback`/`RecordMetadata`, stable across 2.x/3.x, so the adopter's Kafka version wins at runtime). Two halves: (a) **consumer** — `KafkaRecordWorkUnitAdapter`, a `WorkUnitAdapter<ConsumerRecord<*, *>>` (kind `kafka_message`, id from an injectable `messageId` lambda else a UUID) that *is* the message-consumer recipe's hand-rolled adapter applied to Kafka; it writes the `messaging_*` fields as **string literals** (`messaging_system="kafka"`, destination/partition/offset, `messaging_message_id` only when resolvable, `messaging_process_duration_ms`) — literals by policy (not `CanonicalFields` constants), plus the standard check-before-default outcome mapping. No consumer loop runner ships — the poll loop stays the adopter's (024 recipe). (b) **producer** — `Producer<K,V>.withCanonicalLogging()`, a delegating `Producer` (a decorator, deliberately **not** a `ProducerInterceptor`: interceptors are reflectively config-instantiated with no construction seam, and `onAcknowledgement` runs on producer IO threads with no ambient binding) that mirrors the OkHttp capture-at-submit tag pattern — captures `currentCanonicalContext()` on the calling thread at `send`, increments `KAFKA_PRODUCE_COUNT` at submit and adds `KAFKA_PRODUCE_DURATION_MS_TOTAL` / `KAFKA_PRODUCE_ERROR_COUNT` from the wrapped ack callback against the *captured* context. These producer aggregates *are* `CanonicalFields` constants (this module's own contributor fields, like `db_*`/`http_client_*`); contributions go through `increment` only. No work unit bound → pure delegation, zero fields. `api(project(":canonical-log-core"))`.
- `canonical-log-test` — test kit for adopters: `captureCanonicalLine{,Blocking}` run a block through the real work-unit lifecycle and return a `CapturedCanonicalLine` (emit-time field snapshot, `WorkUnit`, `Outcome`, block result/exception — block exceptions are captured, not rethrown, so failure lines are assertable); `testCanonicalLogContext()` + `withBoundCanonicalContext()` for unit-testing contributors against a bare context. Also `RecordingCanonicalAppender` (todo 032) + the public `canonicalFields(event)` flattener — the observe-a-real-booted-stack counterpart to the capture harness: `captureCanonicalLine{,Blocking}` stays the tool whenever the code under test can be driven **in-process**; reach for `RecordingCanonicalAppender` only when the line must come out of an actual container/context on a producer thread (Jetty/Dropwizard, a Spring `@Scheduled` thread, a broker consumer), which is exactly what made hand-rolled `appender.list` reads flaky. `attach()` hooks a `ListAppender` to the `"canonical"` logger (pass a higher logger, e.g. `ROOT`, to also observe sibling loggers through one appender); `awaitLine(where)` polls a defensive snapshot (never the live list, so no `ConcurrentModificationException`), filters the canonical line, and requires **exactly one** match, returning its flattened field map (`canonicalFields` supports both line shapes — logstash `MapEntriesAppendingMarker`, detected reflectively so no encoder dep leaks into main, and MDC `mdcPropertyMap`); `assertNoLine(where)` is the negative form; `events()`/`clear()`/`close()` round it out. The poll is a plain `Thread.sleep` loop — `AutoCloseable`, coroutine-free. Adopted by `CanonicalLogBundleTest` (Dropwizard/MDC) and `CanonicalSchedulingAutoConfigurationTest` (scheduling/marker), which dropped their private await/snapshot helpers. Framework-agnostic beyond `logback-classic` (an `api` main dep — logback is the sink every adopter test uses; still no kotest/JUnit deps in main); wraps the `@DelicateCanonicalLogApi` internals so adopters don't need the opt-in.
- Sample app — exercises everything.

The split between `canonical-log-<lib>` and `canonical-log-<lib>-spring-boot-starter` follows Java ecosystem convention: contributors are framework-agnostic, starters are the integration glue. This keeps the door open for Quarkus/Micronaut starters later without duplication.

## Decisions and rationale

These are the decisions that shaped the design. Don't undo them without a strong reason — most have non-obvious justifications.

**Concurrency model.** The accumulator uses `ConcurrentHashMap` with `merge` for atomic increments. Not a synchronized block, not a single-thread assumption. Parallel fan-out within a request (multiple OkHttp calls in flight, parallel DB queries) needs to contribute correctly. Defensive snapshot at emit time to avoid concurrent modification during JSON serialization.

**Coroutine and virtual thread support.** The accumulator is bound to the current execution context via a `ThreadContextElement` that bridges to a thread-local. This means it works for plain threads, virtual threads, and coroutines uniformly. The bridge is the only piece that needs to know about coroutine internals; everything else is thread-local-flavored.

**Open question: `ScopedValue` (JEP 446).** The current implementation uses `ThreadLocal` + `ThreadContextElement`. On JDK 25, `ScopedValue` is the modern alternative and is a better fit for virtual threads and structured concurrency. The accumulator object itself stays mutable (`ConcurrentHashMap`) regardless — this is purely about how the *binding* propagates. Coroutine interop remains awkward either way: `ScopedValue` doesn't integrate with `CoroutineContext` automatically, so the `ThreadContextElement` bridge stays. The choice is whether the JDK-side simplification (and better virtual-thread story) justifies maintaining two binding mechanisms in parallel. Worth prototyping before v0.2 against the suspend-function sample endpoint, which is the case where the difference would actually show up.

**Outcome model.** Two orthogonal concepts:
- **Lifecycle outcome** (`Outcome.Completed` / `Outcome.Threw(cause)`): the library reports facts about whether the unit ran to completion or threw.
- **Semantic outcome** (`CanonicalLog.markFailed(reason, ...)` / `markDegraded(reason, ...)`): the handler expresses intent.

This is a deliberate departure from the original `Outcome.Success` / `Outcome.Failure(Throwable)` design. The two-axis model handles all the cases — exceptions, `Either`, `Result`, KEEP-0441 rich errors, business-level failures with no exception (e.g. `404 post_not_found`) — uniformly. The HTTP adapter defers to handler-set `error_reason` if present and only fills it in itself for 5xx responses or thrown exceptions.

The presence of `error_reason` without `error_class` is the signal that this was a marked failure rather than an uncaught exception. Worth documenting that pattern for query authors.

**Field naming and types.** OpenTelemetry semantic conventions are the source of truth, with dot→underscore mapping for Loki compatibility (`http.request.method` → `http_request_method`). Loki's LogQL sanitizes/mangles dotted keys; underscores are a technical requirement on this stack, not a stylistic preference.

Suffix conventions:
- Durations: `long` milliseconds, `_ms` suffix (e.g. `http_request_duration_ms`)
- Counts: `long`, `_count` suffix (e.g. `db_query_count`)
- Sums of durations: `_duration_ms_total` suffix (e.g. `db_execution_duration_ms_total`)

**Duration fields are integer milliseconds.** Suffix is `_ms` (durations) or `_duration_ms_total` (sums). Sub-millisecond operations report `0`; that's acceptable because canonical log lines target request-level work, not microbenchmarks. Integer over floating-point because JSON floats interact badly with downstream consumers — Avro schemas, language-specific parser quirks, visual readability of raw log lines. **If sub-ms granularity becomes necessary, add a `_ns` integer field alongside `_ms` rather than changing `_ms` semantics** — additive, non-breaking, and lets operators who care opt in by querying the new field. This is the migration path; do not propose changing `_ms` to `Double` or to a different unit.

**Schema versioning.** Include `canonical_log_version=v1` in every line.

**Type rules.**
- Lists: avoid. Decompose to count fields (`tag_count=3`) or delimited strings.
- Nested objects: never.
- Null: omit from output rather than emit `field=null`.

**Library scope discipline (mechanism, not policy).** The library provides the plumbing. PII redaction, cardinality limits, label promotion, sampling, and which fields are "expensive" are all operator/Detekt concerns, not library concerns. Resist requests to move policy into core.

**Contributors vs handlers.** The dividing line is uniformity:
- Contributors capture what's mechanically uniform across all uses of a library (every JDBC query has a duration; every OkHttp call has a status). LaunchDarkly-style flag instrumentation also qualifies — mechanical, bounded, low-cardinality, universally relevant.
- Handlers capture what's per-operation (`post_id`, `cache_hit`, `comment_count`).

If you can't explain a proposed contributor in those terms, it probably belongs in handler code.

**Exception cost on JVM.** The cost is in stack trace construction at throw time, not access time. Kotlin's `Result` type still wraps `Throwable`, so it doesn't escape this cost. `Outcome.Threw(cause)` carries the throwable but doesn't construct anything extra. Sealed classes / Arrow `Either` are the performant alternatives for expected failures — handler code, not library code.

**Three DSLs to ship; four to skip.**
- Ship: configuration DSL, scoped contribution (`canonical("ns") { ... }`), work unit declaration.
- Skip: contributor-authoring DSL (write a class), assertion DSL (negative assertions are first-class but no DSL on top), field registry DSL (operator concern), builder DSL (constructors are fine).

**OTel positioning.** OpenTelemetry is a data model and transport, orthogonal to the canonical log pattern. Ship `canonical-log-otel` as an optional sink module later. Do not make OTel a core dependency.

**Java 17 bytecode floor** (added 2026-07-04, todo 025). Library modules (`:canonical-log-*`, starters included) target Java 17 bytecode; the JDK 25 toolchain still does the compiling and runs the tests. Spring Boot 4 baselines on 17, so the starters go to 17 uniformly with the framework-agnostic modules — no starter exception was needed. The bytecode floor is a **compatibility contract**: a 17 JVM refuses major-version > 61 class files and Gradle refuses via the `org.gradle.jvm.version` module-metadata attribute, so raising the floor later is a *breaking* change for consumers — it only goes up with a deliberate major-ish release. `-Xjdk-release=17` (Kotlin) / `options.release(17)` (javac) are load-bearing and non-negotiable: with a 25 toolchain, `jvmTarget`/target alone changes the class-file version but still compiles against the 25 class library, so an 18+-only API reference would produce a jar that `NoSuchMethodError`s on a real 17 JVM. `-Xjdk-release`/`release` compile against the 17 API surface (ct.sym), turning any such reference into a compile error — that flag *is* the audit that keeps us honest, so don't drop it when touching the root build. Pinned two ways: `canonical-log-core:BytecodeTargetTest` reads `CanonicalLog.class` and asserts major version 61, and CI's `test-jdk17` job re-runs the library suites on a real JDK 17 launcher via `-Ptest.jdk=17` (the root build's per-library `Test.javaLauncher` override, scoped to libraries because samples target 21 and can't load on 17). Samples stay on JDK 21 target.

**OkHttp wiring uses a customizer, not a `BeanPostProcessor`.** The HTTP filter and JDBC starters auto-wire transparently — Spring controls when their hook fires (filter chain, bean construction). OkHttp is different: an `OkHttpClient` is configured via builders and its interceptors can't be added after `build()`. Three options were considered:

1. **`BeanPostProcessor` that wraps `OkHttpClient` beans.** To add an interceptor we'd have to call `.newBuilder().addInterceptor(...).build()`, returning a *different instance*. Anyone holding a reference to the original by constructor injection still has the uninstrumented one — silent bugs. Rejected.
2. **Auto-provide a default `OkHttpClient` bean.** Most non-trivial apps configure their own client (timeouts, dispatcher, cache); a starter-provided default would either be ignored or conflict. Rejected.
3. **Customizer that adopters apply where they construct the client.** Composes with adopter config and other libraries' customizers, doesn't transform anything behind the user's back, mirrors Spring's own pattern (`RestTemplateCustomizer`, `WebClientCustomizer`). The cost is asymmetry: adopters of the OkHttp starter have one extra step the JDBC and HTTP starters don't require.

The asymmetry is real and called out in the README. Pinned end-to-end by `samples/spring-demo:FullStackEndToEndTest`, which boots the sample against a Testcontainers `mccutchen/go-httpbin` upstream and asserts the customizer-wired call produces `http_client_request_count` in the canonical line.

## Subtle gotchas

These have already bitten or nearly bitten:

- **`withCanonicalLog<T, R>`** needs split type parameters (input vs return). Earlier draft had `<T>` for both, which is wrong.
- **`CanonicalLogContext` constructor is currently public** to make adapter testing easy. Before going public, switch to a `forTesting` factory or `@VisibleForTesting` annotation to keep the door open for changes.
- **Field ordering is non-deterministic** because of `ConcurrentHashMap` iteration. If alphabetical or namespace-clustered ordering matters at emit, sort there. Not a v0.1 concern.
- **`error=true` absent means false** — document for query authors that they need `| error="true"` rather than `| error!="false"`.
- **Field names are published as `CanonicalFields` constants** (added 2026-07-03, todo 007). Every field the library itself writes — the `error`/`error_reason`/`error_class`/`degraded`* markers, the `canonical_log_*` self-diagnostics, `work_unit_id`/`work_unit_kind`/`parent_work_unit_id`/`work_unit_depth`, and the contributor fields (`db_*`, `http_*`, `http_client_*`) — is a `const val` on `CanonicalFields` in core, grouped and KDoc'd with type + semantics. The contributors, adapters, and `CanonicalLogMdc.KEY` reference the constants instead of string literals, so the "constants match what's emitted" assertion is largely tautological; `CanonicalFieldsTest` additionally pins each constant's *value* (the on-the-wire contract — a value rename silently breaks every dashboard) and asserts no two constants alias the same string. Deliberately **not** a field registry or schema DSL — that's an operator/Detekt concern and an explicit anti-goal; handlers still `put` any key they like. `canonical_log_version` is intentionally absent (documented as `v1` in the decisions section but not currently written by any code — no constant for a field nothing emits).
- **Adapter `enrich` wins over the handler for the same key** (precedence, made explicit in todo 007). `enrich` runs *after* the handler block, so an unconditional `ctx.put` there overwrites a handler value under the same key. That's intended for the mechanically-uniform fields (status, durations, identity). The deliberate exceptions are the two "intent" fields — `error_reason` and `cancel_reason` — where `HttpWorkUnitAdapter` checks `ctx.snapshot()[key] == null` before writing its default (`"exception"`/`"server_error"`/`"cancelled"`/`"async_timeout"`), so a handler-set value survives. Custom adapters should follow the same check-before-default pattern for any field a handler is expected to own. Documented on `WorkUnitAdapter.enrich`. **Since todo 023 the full write order for a key is `seed` → handler → `enrich`** (enrich authoritative): the optional `WorkUnitAdapter.seed(ctx, input)` default-method hook runs at work-unit *open*, on the *opening thread*, so both the handler and `enrich` overwrite its values — seed values are defaults, not authority. `seed` exists for **ambient capture, not identity**: request-scoped state that only exists at the start on the opening thread (an MDC `trace_id`/`request_id`, the current span id) and would be gone by `enrich` time (which may run on a servlet `AsyncListener` / scheduling `onStop` thread). Identity still belongs in `describe`. Both core entry points call it — `openCanonicalWorkUnit` after the threadlocal bind (so a logging seed sees `work_unit_id`), and the suspend `withCanonicalLog` synchronously before `withContext` (so it runs on the caller's thread even when the block switches dispatchers). A throwing `seed` is swallowed/WARN'd and recorded via `canonical_log_seed_error*`, mirroring `enrich`; pinned by `WithCanonicalLogTest` + `CanonicalWorkUnitScopeTest`, with the two constants pinned by `CanonicalFieldsTest`.
- **`increment()` on a key holding a non-Long is dropped, not thrown.** Increments run inside contributors embedded in application-critical paths (the JDBC listener's `afterQuery`, the OkHttp interceptor — including its `catch (IOException)` branch, where a throw would *replace* the real exception). Telemetry must never fail the operation it observes, so a put/increment type collision drops the increment and records `canonical_log_type_conflict=true`, `canonical_log_type_conflict_key`, `canonical_log_type_conflict_type` on the line instead. Pinned by `CanonicalLogContextTest`.
- **Spring config metadata is hand-maintained JSON, no annotation processor** (added 2026-07-04, todo 012). Each starter ships `src/main/resources/META-INF/spring-configuration-metadata.json` describing its `canonical-log.*` keys (type, description, default) for IDE autocomplete. Deliberately hand-written (the file IDEs read directly) rather than running `spring-boot-configuration-processor` — with <10 properties total the processor's kapt/KSP build cost isn't worth it. **When you add or rename a `canonical-log.*` property, update that starter's JSON by hand** — nothing generates it. `samples/spring-demo:ConfigurationMetadataTest` guards it: it enumerates every `spring-configuration-metadata.json` on the classpath and asserts the six known `canonical-log.*` keys are present, valid JSON, and each has a type + description (so a malformed file or a missing key fails). Switch to the processor only if the property count grows enough to justify it.
- **The log event's `message` is a human summary, composed by a shared core formatter** (added 2026-07-04, todo 013). `canonicalLineMessage(fields)` in core builds `GET /posts/{id} 200 12ms` (HTTP) or `<work_unit_kind> <work_unit_id>` (non-HTTP), with an ` error=<reason>` hint when `error=true`. It lives in **core**, not the writer, deliberately: there are two emit sites (the HTTP filter and the scheduling starter) and they must produce identical text. As of todo 020 both now share the one `LogstashCanonicalLineWriter` (moved to `canonical-log-logstash`), so the core formatter is what keeps its default `message` identical to any other `CanonicalLineWriter` a custom sink might use. `LogstashCanonicalLineWriter` takes an overridable `message: (Map) -> String` param (default `::canonicalLineMessage`; pass `{ "canonical" }` to restore the old literal). The message deliberately contains no SLF4J `{}` placeholders (route templates use `{id}`, not empty braces), so the `StructuredArguments` argument isn't consumed by placeholder substitution — the fields still attach. Query authors must never parse `message`; every value in it is also a structured field. Non-HTTP units only get `kind`+`id` because entry-point-specific fields (a job's `job_name`) aren't core-known; a custom writer/formatter can do better. Pinned by `CanonicalMessageTest` (core), plus message assertions in `CanonicalLogFilterTest` and `CanonicalSchedulingAutoConfigurationTest`.
- **Blocking lifecycle is an open/close primitive shared by three sites** (added 2026-07-03). `openCanonicalWorkUnit(adapter, input)` binds the context (threadlocal + MDC, records nesting) and returns a `CanonicalWorkUnitScope`; the scope's `outcomeFor(throwable?)` classifies the terminal outcome, `enrich` runs the adapter under the swallow-and-record guard, `unbind` restores the previous binding, `emit` publishes under swallow-and-warn. `withCanonicalLogBlocking` is now a thin closure over it, the servlet filter drives it directly (it must `unbind` on the request thread in `finally` but `enrich`/`emit` later from an `AsyncListener`, which the closure form can't express), and the scheduling starter's observation handler is the third caller. **Since todo 024 the audiences are split.** `openCanonicalWorkUnit` + `CanonicalWorkUnitScope` **graduated out of `@DelicateCanonicalLogApi` — they are now public API**, because an adopter wiring their own consumer/listener/job-runner framework *is* an entry-point author, which is who this lifecycle is for (the worked message-consumer recipe is `docs/recipes/message-consumers.md`, pinned by `canonical-log-test:MessageConsumerRecipeTest`). Their KDoc now carries the explicit numbered invariants (unbind-exactly-once in `finally` on the opening thread; enrich-before-emit, each at most once, may run later/elsewhere; emit-exactly-once is the caller's job via an `AtomicBoolean` guard — `CanonicalLogAsyncEmitListener` is the worked example; else use `withCanonicalLogBlocking`). What stays `@DelicateCanonicalLogApi` is the **binding internals that can corrupt the thread binding**: the `CanonicalLogContext` constructor, `bindCurrentCanonicalContext`/`currentCanonicalContext`'s mutating partner, `CanonicalLogMdc.install/restore`, and `runCanonicalHttpRequest` (servlet). Because `openCanonicalWorkUnit`/`unbind` themselves drive those internals, they carry a *local* `@OptIn(DelicateCanonicalLogApi::class)` at their own definition — public signature, delicate body. **The suspend path (`withCanonicalLog`) deliberately does NOT use this** — its binding is the `CanonicalLogElement` `ThreadContextElement`'s job, so the extraction never touches the coroutine bridge (the property/bridge tests were out of the blast radius and stayed green). The three sites previously hand-rolled the same ~25 lines (core-private `runEnrich`/`safeEmit`/`outcomeOf` plus a re-inlined copy in the filter); consolidating removed that divergence. Pinned by `CanonicalWorkUnitScopeTest` (direct) plus every blocking/filter test (indirect).
- **Telemetry failures never replace the block's result.** The `increment()` principle extended to the tail of the lifecycle: core wraps `adapter.enrich` and `emit` in catch-`Exception` (`Error`s still propagate — JVM-fatal states shouldn't be papered over). The failure is WARN-logged once to the `io.github.alexhumphreys.canonicallog` slf4j logger (message includes the work unit id); enrich failures are additionally recorded on the line via `canonical_log_enrich_error=true` / `canonical_log_enrich_error_class`. The block's result is returned / its exception rethrown as normal, and the threadlocal is restored regardless. The starter's filter applies the same policy to its own `adapter.enrich` / `CanonicalLogSampler` / `CanonicalLineWriter` calls: a throwing sampler fails open (line still written), a throwing writer drops the line with the WARN as the only record. Pinned by `WithCanonicalLogTest` (blocking + suspend emit/enrich cases) and `CanonicalLogFilterTest`.
- **Timestamps in the sample default to local time with offset** (e.g. `+02:00`). Configure Logstash encoder to UTC via `<timeZone>UTC</timeZone>` in `logback-spring.xml` for the sample.
- **Nested work units: inner shadows outer** (defined 2026-07-02, was previously undefined). Opening a unit inside an active one routes ambient contributions to the inner accumulator only; the outer resumes receiving contributions when the inner closes. Each unit emits its own line, inner first. A nested unit's line carries `parent_work_unit_id` with the *immediate* parent's id (reconstruct deeper chains by joining lines) and `work_unit_depth` (1 = opened inside a top-level unit, 2 inside that, ...; a direct "how nested is this?" query handle — deep values usually mean something complicated and bug-prone); top-level lines omit both fields (absent means depth 0). No aggregation of inner fields into the outer line — deliberately out of scope. The contract is identical for `withCanonicalLogBlocking` and `withCanonicalLog` nested in any combination; pinned by `NestedWorkUnitTest`.
- **Multi-line emit for long jobs is rejected** by design. Long jobs should be decomposed into multiple work units, each with its own canonical line. Don't add a `flush()` to the accumulator.
- **`withCanonicalLog` block has a `CoroutineScope` receiver** — `suspend CoroutineScope.(CanonicalLogContext) -> R`, not `suspend (CanonicalLogContext) -> R`. The receiver is load-bearing: without it, bare `async { ... }` inside the block resolves against the *outer* scope (e.g. `runTest`'s `TestScope`, `runBlocking`'s scope, or whatever wraps the call) and the new coroutine inherits the outer context, which has no canonical element. Contributions from inside the async coroutine are silently lost. The fix is the receiver, not `CopyableThreadContextElement` — that was investigated and didn't address the actual cause. Found and fixed during bridge bulletproofing; pinned by `BridgeContractTest` cases 3 and 8. The block also runs inside an inner `coroutineScope` (the receiver *is* that inner scope), so structured children the block launches but never joins are awaited before enrich/emit — without it, enrich/emit ran inside the `withContext` lambda, before `withContext` joins its children, so un-joined launches raced with emit and a failing child surfaced to the caller after a `Completed` line had already been emitted. Pinned by the two "un-joined structured child" cases in `WithCanonicalLogTest`.
- **Nested `withCanonicalLog` calls compose via `ThreadContextElement`'s default merge** — innermost-`Key` wins, the outer accumulator stops receiving contributions while the inner is open, and resumes once the inner exits. This *is* the pinned inner-shadows-outer contract (see the nested-work-units entry above), implemented by the default merge on the suspend path and by explicit previous-binding save/restore on the blocking path. `parent_work_unit_id` and `work_unit_depth` (parent's depth + 1, read back from the parent's own field) are detected in both entry points by reading the threadlocal at open time — the threadlocal, not the coroutine context, because it always reflects the *innermost* active unit on the current thread (a blocking inner unit under a suspend outer is only visible there). Pinned by `NestedWorkUnitTest`; no `CopyableThreadContextElement` needed.
- **Plain thread hops need the propagation helpers.** The coroutine bridge doesn't cover `ExecutorService`/`@Async`/`CompletableFuture.supplyAsync` — the worker thread has no binding, so contributions are silent no-ops. `ContextPropagation.kt` provides `Runnable`/`Callable`/`Executor.propagatingCanonicalContext()` (capture at wrap/submit time on the submitting thread, bind/restore around the task). The work unit still does not *wait* for such tasks — join before the block returns or contributions are cut off at emit. These wrappers still can't fix OkHttp `enqueue()` (queued calls are promoted from OkHttp-owned threads, so capture-at-submit grabs the wrong context) — that path is handled by a request tag instead; see the OkHttp `enqueue()` entry below. Pinned by `ContextPropagationTest`.
- **OkHttp `enqueue()` is supported via a request tag** (added 2026-07-03, todo 008). `OkHttpCanonicalInterceptor` resolves the work unit tag-first: `request.tag(CanonicalLogContext::class.java)` before the calling-thread binding (`currentCanonicalContext()`). Synchronous `execute()` needs nothing (the interceptor runs on the caller's bound thread); async `enqueue()` runs the interceptor on OkHttp dispatcher threads with no binding, so the caller opts in by tagging the request at build time with `Request.Builder().withCanonicalContext()` (a module extension in `RequestTagging.kt`; its default arg captures `currentCanonicalContext()` on the calling thread, where the binding is live). Untagged `enqueue()` stays a no-op — the tag is opt-in and adds no field. The interceptor was refactored off the ambient `CanonicalLog` object to resolve a `CanonicalLogContext?` once and call `ctx?.increment(...)`; the tag path needs no `@OptIn` because `currentCanonicalContext()` is public API. The work unit still doesn't *wait* for the queued call — join it (e.g. block on a latch inside the work-unit block) before emit or the contributions miss the snapshot. Deliberately skipped the optional `Call.Factory` auto-tagging wrapper — the explicit tag earns its keep, an auto-wrapper didn't justify the extra API surface. Pinned by `OkHttpCanonicalInterceptorTest` (tagged enqueue contributes; untagged is a no-op; tag captured outside a work unit attaches nothing and doesn't NPE).
- **Kafka producer acks that land after emit are dropped** (added 2026-07-07, todo 026). `Producer.withCanonicalLogging()` captures the work unit on the calling thread at `send` and increments `kafka_produce_count` synchronously; the duration and any error count are added from the ack callback (which runs on producer IO threads), against the *captured* context. If the ack fires after the work unit emitted its line, the snapshot-at-emit cutoff means those contributions are silently absent from that line — identical to OkHttp `enqueue()` and orphaned coroutine launches. A work unit that must account for its sends should `flush()` / block on the returned `Future` before returning. The decorator was chosen over a `ProducerInterceptor` deliberately (reflective config instantiation, no ambient binding on the ack thread — see the module-layout entry). Pinned by `CanonicalLoggingProducerTest` (the after-emit case asserts the count lands but the duration doesn't, and the late ack neither throws nor mutates the emitted line).
- **MDC correlation rides the threadlocal binding** (added 2026-07-03, todo 006). `CanonicalLogMdc` mirrors the active work unit's id into slf4j MDC under `work_unit_id` (same name as the line's field), so ordinary log lines join to their canonical line with one equality query. Every binding site pairs `install` (returns the displaced MDC value) with `restore`: `withCanonicalLogBlocking`, the `CanonicalLogElement` bridge (its saved per-dispatch state is now a `Restore(context, mdcValue)` pair — MDC follows dispatcher switches with no `MDCContext` pairing), the `propagatingCanonicalContext()` wrappers, and `CanonicalLogFilter`. Deliberately *not* folded into `bindCurrentCanonicalContext`: its restore-by-rebinding contract can't carry the previous MDC string, so entry-point authors using the delicate API pair it with `CanonicalLogMdc.install/restore` themselves (documented on both). Nesting shows the innermost id, matching where contributions go. Opt-out is the process-wide `CanonicalLogMdc.enabled` switch — set once at startup (the starter binds it from `canonical-log.http.mdc-enabled`; it's global, not per-filter, because the coroutine bridge and hop wrappers would otherwise keep writing MDC for shops that manage MDC themselves). The logstash encoder includes MDC by default, so the sample's ordinary lines carry the field with no logback change. Pinned by `MdcCorrelationTest` (core) and the MDC cases in `CanonicalLogFilterTest` / `CanonicalLogAutoConfigurationTest`.
- **Blocking entry + suspend body needs `withCanonicalCoroutineContext`** — the blocking variant only sets the threadlocal. A suspend body that switches dispatchers (e.g. `withContext(Dispatchers.IO)`) lands on a thread where the threadlocal is empty. Wrap the body in `withCanonicalCoroutineContext { ... }` (which reads the active threadlocal and installs a `CanonicalLogElement` in the coroutine context) before any dispatcher switch.
- **Async servlet dispatch breaks the naive filter pattern** — `chain.doFilter` returns *immediately* for async-dispatched handlers (suspend controllers, `Callable`/`DeferredResult`, SSE). A filter that emits in its `finally` block emits before the handler has run. The lifecycle checks `request.isAsyncStarted` and defers emit to an `AsyncListener` for async paths. Since todo 021 this lifecycle lives in **one place** — `runCanonicalHttpRequest` in `canonical-log-servlet` — and both filters (the Spring `CanonicalLogFilter` and the framework-neutral `CanonicalLogServletFilter`) are thin callers of it; the single-emit `CanonicalLogAsyncEmitListener` and `AsyncTimeoutCancellationException` moved there too. Change the async behaviour in that helper, not per-filter. Found during suspend-endpoint integration; pinned by `CanonicalLogFilterTest` (Spring), `CanonicalLogServletFilterTest` + `CanonicalLogAsyncEmitListenerTest` (servlet), and the `ab` load test.
- **Spring suspend handlers need `kotlin-reflect` and `kotlinx-coroutines-reactor` on the classpath** — without them, `KotlinDetector.isSuspendingFunction` returns false and Spring tries to invoke the suspend method via plain reflection, which NPEs on the missing `Continuation`. Document this in the umbrella starter's docs once we have any.
- **Uncaught exception status capture is filter-stage-bound.** When a controller throws an unhandled exception, Tomcat's outer valve maps it to 500 *after* `CanonicalLogFilter` unwinds. The filter sees `response.status` as still 200 at the moment of catch. The adapter compensates: when `Outcome.Threw` and `response.status < 500`, it overrides `http_response_status_code` to 500 to match what the client actually receives. This is heuristic — if a custom error handler maps to a different 5xx (e.g. 503), the canonical line will report 500 instead. Pinned by `HttpWorkUnitAdapterTest`; documented as a known approximation.
- **Cancellation is a first-class outcome, not an error** (defined 2026-07-02, was previously undefined). A block that terminates with `CancellationException` produces `Outcome.Cancelled(durationMs, cause)` in both entry points. Classification is by exception type (`is CancellationException`), deliberately not by inspecting the job — a CE leaked from an unrelated child (`await()` on an externally cancelled `Deferred`) is still reported as Cancelled. The CE is always rethrown after enrich/emit, never swallowed, so structured concurrency stays intact; this works in the suspend variant because enrich/emit are non-suspending — there is no suspension point between catching the CE and rethrowing it for the already-cancelled coroutine to abort at. Line shape: `cancelled=true` + `cancel_reason` (adapter default `"cancelled"`; a pre-set value wins, mirroring the `error_reason` policy), **no** `error=true`/`error_class` — client disconnects and timeouts must not pollute error rates. `HttpWorkUnitAdapter` reports `http_response_status_code=499` (nginx's "client closed request" convention) unless the response already carried a ≥400 status; the on-the-wire status after an async timeout is container-dependent (Tomcat often writes a 500 error page after the filter unwinds) — the line deliberately reports the cancellation, not the container's error page. `CanonicalLogFilter`'s `AsyncListener.onTimeout` no longer synthesizes a `TimeoutException`/`error=true`: it synthesizes an internal stackless `AsyncTimeoutCancellationException` so the same CE→Cancelled mapping applies and the line gets `cancel_reason="async_timeout"` — the servlet and suspend entry points tell the same story. In-flight contributions: whatever is in the accumulator at emit time lands (same snapshot cutoff as everything else; documented, not engineered). One inherent race on the suspend path: a caller's job cancelled *after* the block completes but before `withContext` returns yields a `Completed` line while the caller sees a CE — the line reflects how the block itself terminated. Adding the `Outcome.Cancelled` subtype is source-breaking for exhaustive `when`s over `Outcome` in adopter adapters (fine pre-1.0; changelog-worthy when releases start). Pinned by `WithCanonicalLogTest` (blocking + suspend cancellation cases), `HttpWorkUnitAdapterTest`, `CanonicalLogFilterTest` (timeout path), and `CanonicalLogFilterAsyncPropertyTest`.
- **Virtual threads work end-to-end.** Verified: with `spring.threads.virtual.enabled=true` in `application.properties`, the entire test suite (67 tests) passes, the suspend endpoint runs entirely on virtual threads (`Thread.currentThread().isVirtual == true` on every request entry), and `ab -n 5000 -c 50 -k http://localhost:8080/suspend/posts/1` produces 5000 lines with zero field bleeding. The sample app currently has virtual threads enabled by default. The bridge, filter, accumulator, and contributors all behave identically on virtual and platform threads — the threadlocal abstraction abstracts over both uniformly.
- **JDBC starter replaces `DataSource` beans with proxies.** `JdbcCanonicalBeanPostProcessor` runs at `Ordered.LOWEST_PRECEDENCE` so it wraps the outermost `DataSource` proxy. Two adopter-visible consequences: (a) injecting by concrete type (`@Autowired private val ds: HikariDataSource`) fails with `BeanNotOfRequiredTypeException` — adopters must inject `DataSource` instead; (b) when another `datasource-proxy` user is on the classpath we add ourselves to the existing chain rather than re-wrap, but if a *non-`datasource-proxy`* tracing wrapper sits above us we proxy that wrapper, which means our `db_execution_duration_ms_total` includes its overhead. Both behaviours are deliberate (pinned by `JdbcCanonicalAutoConfigurationTest`); the opt-out is `canonical-log.jdbc.enabled=false`. The BPP also skips beans that already have a `JdbcCanonicalListener` anywhere down their delegation chain (walked via `ProxyDataSource.getDataSource` plus duck-typed `getTargetDataSource()`), so an outer `DelegatingDataSource`-style bean over an already-wrapped inner bean doesn't double-count — the cost is that the outer wrapper's overhead is then *not* in `db_execution_duration_ms_total` for such stacks. `AbstractRoutingDataSource` target maps are not walked; a router bean over target beans still double-counts.
- **Multiple `DataSource` beans aggregate into one canonical line.** The BPP wraps every `DataSource` it sees, and all of them write to the same active accumulator. So `db_query_count` for a request that touches both a `primary` and a `replica` is the sum across both. There is no per-datasource namespacing today (`db_query_count_primary` etc.). Decide whether to namespace before adopters with read-replica setups depend on the aggregated shape.
- **Spring Boot's logback initialization clears `ListAppender` instances attached before `SpringApplication.run`.** Tests that capture canonical lines via a `ListAppender` on the `"canonical"` logger must attach the appender *after* `SpringApplication.run` returns, not before — otherwise logback re-reads `logback-spring.xml` during boot and the test sees zero events. Pinned by `samples/spring-demo:FullStackEndToEndTest` (which has an inline comment explaining this so the next person doesn't re-discover it).
- **JDBC per-execution mean durations are reliable in integer ms; per-statement durations are not.** `db_execution_duration_ms_total / db_execution_count` gives mean per-round-trip latency, which is typically multi-ms even when individual statements aren't. Dividing by `db_query_count` instead would *under*-report per-statement latency for batched executions, because `datasource-proxy` charges duration once per `afterQuery` regardless of statement count. This is why the field rename happened (the old `db_query_duration_ms_total` was the worst of both worlds); document it here so the next person doesn't try to "fix" it back.

## Testing

**Framework.** Kotest. Don't introduce JUnit-style tests for new code. One deliberate
exception: `canonical-log-core/src/test/java/JavaErgonomicsTest.java` is plain JUnit Jupiter
because its entire purpose is to pin the Java-caller ergonomics (`@JvmStatic` statics without
`INSTANCE`, `@JvmOverloads` on `increment`, `Map` overloads of `markFailed`/`markDegraded`
instead of `kotlin.Pair` varargs, `withCanonicalLogBlocking` via Java lambdas) — it has to be
Java, and Kotest isn't callable from Java. Don't port it to Kotlin.

**What to test at which level.**

- *Per-contributor tests* live in the contributor module and verify the contributor produces the expected fields given a controlled invocation (e.g. running an OkHttp call through the interceptor in isolation, asserting `http_client_request_count=1` and a sensible duration). These should not require Spring or Boot.
- *Per-starter tests* live in the starter module and verify the auto-configuration wires correctly — the bean is registered, the contributor is active, fields appear on a real request. Spring Boot test slices are appropriate here.
- *End-to-end tests* live in the sample app and verify the full pipeline: a real HTTP request produces one canonical line with the expected fields from all contributors plus handler-supplied fields. The example log lines committed inline in the top-level `README.md` are effectively the spec for these tests.

**Negative assertions are first-class.** When testing a contributor or handler, assert what's *not* in the output as well as what is. `hasNoField(name)` and `hasNoFieldMatching(regex)` are the helpers. Common cases:
- Asserting that a contributor doesn't leak request bodies, headers, or query params it wasn't asked to capture.
- Asserting that `error=true` is absent on success paths (rather than asserting `error=false`, which would be wrong — the convention is "absent means false").
- Asserting that `error_class` is absent when a handler called `markFailed` without throwing.

These keep contributors honest about scope and prevent accidental field creep, which is the failure mode that turns canonical logs into a cardinality nightmare.

**Concurrency validation.** Any change to the accumulator, the `ThreadContextElement` bridge, or the binding mechanism (including a `ScopedValue` migration) needs a load test against the sample app. The standard check:

```
ab -n 1000 -c 10 http://localhost:8080/posts/1
jq -r 'select(.logger_name=="canonical") | .post_id' canonical.log | sort | uniq -c
jq -r 'select(.logger_name=="canonical") | .db_query_count' canonical.log | sort | uniq -c
```

For a single-endpoint test, every line should have the same `post_id` and the same steady-state count values. Distribution = field bleeding between requests = bug. Line count should match the request count exactly; off-by-one means a lifecycle bug.

**Concurrency testing approach for unit tests.** The load test catches gross failures end-to-end; unit tests catch subtle failures deterministically. The conventions:

- **Kotest** is the test framework. `kotlinx-coroutines-test` for suspending code, `kotest-property` for invariants under random schedules.
- **`runTest { }`** for any test involving coroutines or dispatchers. Gives deterministic execution with a virtual scheduler — no flaky tests from scheduling variance.
- **Property-based tests** for the accumulator's concurrency invariants. Generate N concurrent contributions, run them via `coroutineScope { repeat(N) { launch { ... } } }`, assert that totals equal sums and no contributions are lost. The schedule varies between runs, which catches bugs that single-shot tests miss.
- **`continually { }`** for negative timing assertions ("no canonical line should appear during this window because the work unit hasn't finished").
- **jcstress** is available if something forces the question of exhaustive schedule exploration, but don't reach for it without evidence the standard tests miss something. Overkill for the current design.

**The suspend-function path is verified.** A suspend controller (`/suspend/posts/{id}`) is wired into the sample app and was exercised under `ab -n 1000 -c 10`. All 1000 canonical lines emit with the expected fields and zero field bleeding (every line has the same `post_id`, `db_query_count`, `http_client_request_count`, status code).

**Bridge contract.** What the `ThreadContextElement` bridge guarantees, derived from the contract tests in `BridgeContractTest` and the property tests in `AccumulatorPropertyTest`:

1. **Same-dispatcher contributions** — fields put inside the `withCanonicalLog` block, on the entering dispatcher, are in the snapshot.
2. **`withContext` switch** — fields put inside `withContext(otherDispatcher)` are in the snapshot, on whichever thread that dispatcher uses.
3. **`async { }.await()`** — fields put inside an `async` block are in the snapshot after `await()` returns. **Important precondition:** the `async` must resolve against the `withCanonicalLog` block's `CoroutineScope` receiver — which it does naturally, because the block signature is `suspend CoroutineScope.(CanonicalLogContext) -> R`. If you call `async` from a place where a different `CoroutineScope` is the receiver (e.g. a top-level extension function called from inside the block, or `runTest`'s `TestScope` shadowing), the new coroutine inherits the wrong context and contributions are lost. See "Subtle gotchas" below.
4. **Parallel `async` fan-out** — `coroutineScope { async; async; ... awaitAll() }` works; per-branch increments sum correctly. The accumulator's `ConcurrentHashMap.merge` makes this safe.
5. **Nested `withContext` switches A→B→A** — context survives the round trip; contributions in every layer land.
6. **Orphaned `launch` outliving the parent** — by design, contributions made *after* `withCanonicalLog` has emitted are NOT in the emitted snapshot (the snapshot is a defensive copy taken at emit time). The launched coroutine still executes; its puts are silently swallowed for the purposes of *that* canonical line. This is the intentional cutoff for `launchDetached`-style fire-and-forget. The accumulator object remains valid; nothing throws.
7. **Sharing within the work unit** — child coroutines (launches and asyncs) share the parent's accumulator; their contributions are visible to the parent before emit. This includes structured children the block never joins: the block runs in an inner `coroutineScope`, so un-joined launches complete before the outcome is computed and the line is emitted (and a failing un-joined child makes the work unit report `Outcome.Threw`). Pinned by `WithCanonicalLogTest`.
8. **Blocking-entry → suspend bridge** — when the work unit is opened by a blocking entry point (e.g. the servlet filter calling `withCanonicalLogBlocking`) and a suspend body needs to bridge it into a coroutine context, use `withCanonicalCoroutineContext { ... }`. This reads the active threadlocal and lifts it into the coroutine context for downstream `withContext` switches. No-op if no work unit is active.

**Async servlet dispatch is load-bearing.** Suspend controllers in Spring MVC are dispatched asynchronously: `chain.doFilter` returns *before* the handler completes. The lifecycle detects this via `request.isAsyncStarted` and registers an `AsyncListener` to defer emit until completion. The single-emit invariant is enforced by an `AtomicBoolean`. This lifecycle is `runCanonicalHttpRequest` (in `canonical-log-servlet`, todo 021) — shared by the Spring filter and the plain `CanonicalLogServletFilter` — so change it once, there. Preserve: (a) emit-exactly-once across `onComplete`/`onError`/`onTimeout`, (b) sync-handler emit-inline, and (c) thread-local cleanup before the filter returns regardless of sync/async path.

## Anti-goals

These have been considered and rejected. Don't add them without a new and compelling argument:

- **Contributor-authoring DSL.** Writing a `WorkUnitAdapter` or contributor as a regular class is fine. A DSL would obscure the lifecycle.
- **Assertion DSL.** Negative assertions are first-class API but there's no need for a fluent `expect { ... }` layer.
- **Field registry DSL.** Operators decide what fields to allow at the Detekt or pipeline level, not the library.
- **Builder DSL** for `WorkUnit` etc. Constructors are clearer.
- **Multi-line emit for long-running jobs.** Decompose into multiple work units instead. The "one line per unit of work" invariant is load-bearing.
- **Hiding distributed complexity behind a DSL.** Starters wire resilience patterns (Resilience4j, DLQ, retries) without hiding service boundaries. No "call this remote service like it's local" abstractions.
- **Lists or nested objects in field values.** Decompose to count fields or delimited strings; objects never.
- **PII redaction in core.** Operator concern. The library should make it easy to plug in, not opinionated about what's sensitive.
- **Sampling in core.** Same reasoning. The starter exposes the hook (`CanonicalLogSampler`, a post-enrich emit predicate — see the module layout section); core stays simple, and rate/policy decisions live in adopter code.
- **OTel as a core dependency.** Optional sink module only.
- **Field-name renaming hook.** Field names are deliberately fixed across the library — `http_request_method`, `db_execution_count`, etc. are the contract. No `FieldNameMapper` interface, no per-app config to rename `db_query_count` to `database_queries_total`. Reason: canonical log lines are designed to be queried with a single shape across services in the same observability stack; per-service renames would let two services emit semantically identical fields under different names, which defeats the point. Workarounds for adopters who need different names downstream: a logback encoder, a Logstash filter, or a thin transformation step in the log pipeline — all of which keep the rename out of the library and at the edge where the operator already controls the schema.

## Working with this codebase

A few notes for an agent picking this up:

- The decisions section above is load-bearing. If you find yourself wanting to change `Outcome`, the field naming, the contributor/handler split, or the "mechanism not policy" rule, stop and ask first — the rationale is non-obvious and was hard-won.
- The accumulator's concurrency story is the riskiest piece. Any change there needs a load test (`ab -n 1000 -c 10` against the sample) to verify no field bleeding.
- The `ThreadContextElement` bridge has only been verified on the blocking servlet path. The suspend-function path is on the to-do list above; until that's done, treat coroutine support as theoretical.
- Negative assertions (`hasNoField`, `hasNoFieldMatching`) are first-class. When adding behavior, add tests that assert what's *not* in the output as well as what is.
- Resist scope creep into the anti-goals list. If a feature request seems compelling, write down why before adding it — the list above exists because every item on it seemed compelling at some point.
